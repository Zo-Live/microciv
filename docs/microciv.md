## 1. 项目定位与目标

### 1.1 项目名称

**MicroCiv：微型文明**

### 1.2 项目定位

MicroCiv 是一个在终端中运行的微型文明模拟游戏。项目聚焦于“地图生成、资源经营、城市扩张、道路网络、科技研发、AI 决策与实验记录”这一完整闭环，目标是构建一个：

1. 可运行
2. 可扩展
3. 可比较
4. 可用于课程实验分析

的规则驱动策略环境。

### 1.3 当前目标

当前版本的正式目标包括：

1. 实现完整的正方形网格地图规则
2. 实现城市、道路、建筑、科技、资源与评分系统
3. 实现完整的 curses 终端交互流程
4. 提供 `Baseline` 与 `Random` 两种自动播放 AI
5. 提供可导出的 Records 数据，用于后续分析不同 AI 的效果与速度

### 1.4 AI 对照目标

项目内置两类基础策略：

1. `Baseline`
   采用规则化启发式策略，用于作为主要基准
2. `Random`
   在合法动作集合中随机选择动作，用作对照组

两者都必须能够在完整对局中稳定输出合法动作，并为 Records 留下可比较的表现与耗时数据。

## 2. 总体设计原则

### 2.1 规则层与表现层分离

系统采用“游戏逻辑层”与“界面表现层”解耦设计：

1. 逻辑层负责状态、合法性判断、动作执行、结算、评分、AI 决策
2. curses 界面层负责绘制、鼠标命中、快捷键、页面切换、Records 展示
3. AI 只依赖逻辑层，不依赖具体界面

### 2.2 鼠标优先，键盘辅助

虽然项目基于 curses 实现，但交互方式明确为：

1. **主要依靠鼠标操作**
2. 键盘快捷键作为补充与提效手段
3. 所有核心按钮、地块、科技项、建筑项、Records 条目都应支持鼠标点击
4. Records 页面支持鼠标滚轮滚动

### 2.3 AI、播放速度与地图难度解耦

Autoplay 相关配置拆分为三个独立维度：

1. **AI Policy**
   - Baseline
   - Random
2. **Playback Mode**
   - Normal
   - Speed
3. **Map Difficulty**
   - Normal
   - Hard

这三个维度必须在界面、Records 和 CSV 中独立记录，避免后续实验统计混淆。

### 2.4 视觉实现原则

1. 所有面状图形都使用 Unicode block elements 绘制
2. 地图、城市、道路和高亮状态都以块字符与颜色组合表示
3. 普通说明文本使用终端默认字体
4. 界面优先保证稳定性、清晰度和可点击性

## 3. 地图、坐标与地形设定

### 3.1 地图形状与坐标系

1. 地图采用**正方形网格地图**
2. 地图大小参数 `map_size` 表示边长
3. 正式支持范围为 `12 ~ 24`
4. 默认地图大小为 `18`
5. 坐标使用 `(x, y)` 表示，其中左上角为 `(0, 0)`
6. 地图总格数公式为 `map_size * map_size`

### 3.2 邻接规则

项目中显式区分两类邻接：

#### 3.2.1 八邻域（Moore 邻域）

1. 城市采集资源使用八邻域
2. 即城市可收集周围 `3 x 3` 范围内除自身外的 8 个格子

#### 3.2.2 四邻域（Von Neumann 邻域）

1. 道路与城市的连通关系使用四邻域
2. 只有共享边的格子可修路连通
3. 对角线不能直接形成道路连接

### 3.3 地形类型

地图包含以下五类基础地形：

1. 平原 `plain`
2. 森林 `forest`
3. 山地 `mountain`
4. 河流 `river`
5. 荒地 `wasteland`

此外还有两类占据状态：

1. 城市 `city`
2. 道路 `road`

### 3.4 地形规则

#### 3.4.1 平原

1. 可建城
2. 可修路
3. 产出 `+2 food / turn`

#### 3.4.2 森林

1. 可建城
2. 可修路
3. 产出 `+2 wood / turn`

#### 3.4.3 山地

1. 可建城
2. 可修路
3. 产出 `+2 ore / turn`

#### 3.4.4 河流

1. 不可建城
2. 可修路
3. 产出 `+1 food / turn`
4. 额外产出 `+1 science / turn`

#### 3.4.5 荒地

1. 不可建城
2. 可修路
3. 不提供资源

## 4. 地图生成规则

### 4.1 初始地图生成约束

初始地图必须满足：

1. 只生成基础地形，不生成城市与道路
2. 至少存在 1 个平原格
3. 荒地不能与河流相邻
4. 地图具备基本可玩性与资源分布多样性

### 4.2 总体生成流程

地图生成采用固定流程：

1. 枚举正方形网格全部坐标
2. 选择若干区域种子
3. 按距离划分区域
4. 为各区域分配主地形
5. 注入少量次级地形斑块
6. 对局部地形做平滑处理
7. 叠加一条或两条河流
8. 修补河流邻接荒地
9. 执行质量检查与必要修补

### 4.3 地图难度差异

#### 4.3.1 Normal

1. 平原比例更高
2. 可建设地形比例更高
3. 河流数量为 1

#### 4.3.2 Hard

1. 平原比例更低
2. 山地与荒地比例更高
3. 河流数量为 2

## 5. 资源、城市与道路规则

### 5.1 城市采集规则

1. 城市从八邻域中的未被占据格子收集资源
2. 一个地块只会将某种资源归属于一个城市
3. 若多个城市竞争同一资源，则按以下顺序决定归属：
   1. 对应建筑数量更多者优先
   2. 建城更早者优先
   3. 坐标更靠前者优先

### 5.2 城市基础消耗

1. 每座城市每回合固定消耗 `4 food`

### 5.3 断粮规则

若某个网络在本轮结算开始时食物小于等于 0，则：

1. 仍然可以获得食物产出
2. 其它资源产出停工
3. 建筑中的非食物产出同样停工

### 5.4 覆盖收益

建城或修路覆盖地块时，一次性获得资源：

1. 覆盖平原：获得 `8 food`
2. 覆盖森林：获得 `8 wood`
3. 覆盖山地：获得 `8 ore`
4. 覆盖河流：获得 `8 food + 8 science`

### 5.5 道路与网络

1. 道路必须与已有城市或道路四向相邻
2. 城市、道路与河流共同构成网络可通行空间
3. 通过道路与河流连接起来的城市属于同一网络
4. 网络共享资源与已解锁科技

### 5.6 河流修路额外消耗

在河流上修路时，所在相邻网络必须额外支付：

1. `3 wood`
   或
2. `2 ore`

支付优先使用木材，若木材不足则使用矿石。

## 6. 建筑、科技与评分

### 6.1 科技树

1. `Agriculture`：消耗 `6 science`，解锁农场
2. `Logging`：消耗 `12 science`，解锁伐木场
3. `Mining`：消耗 `20 science`，解锁矿场
4. `Education`：消耗 `20 science`，解锁图书馆

### 6.2 建筑

1. 农场：消耗 `5 wood`，产出 `+3 food / turn`
2. 伐木场：消耗 `5 wood`，产出 `+2 wood / turn`
3. 矿场：消耗 `6 wood + 4 ore`，产出 `+2 ore / turn`
4. 图书馆：消耗 `6 wood + 4 ore`，产出 `+2 science / turn`

### 6.3 建筑上限

每座城市最多容纳 `6` 个建筑。

### 6.4 评分函数

总分 `S` 为：

`20 * 城市数 + 15 * 建筑数 + 30 * 科技数`

再加上资源结余折算：

1. `food // 10`
2. `wood // 5`
3. `ore // 5`
4. `science // 20`

该评分函数鼓励：

1. 稳定扩城
2. 资源转化为建筑
3. 持续推进科技
4. 控制无意义囤积

## 7. 回合流程与动作规则

### 7.1 合法动作

每回合玩家或 AI 可执行以下动作之一：

1. 建城
2. 修路
3. 造建筑
4. 研究科技
5. Skip

### 7.2 每回合执行顺序

一次动作成功执行后，系统按以下顺序推进：

1. 应用动作本身
2. 重算网络
3. 重算资源归属
4. 结算地形产出
5. 结算建筑产出
6. 扣除城市粮食消耗
7. 更新分数
8. 更新回合计数
9. 达到回合上限则结束游戏

### 7.3 非法动作处理

若动作非法：

1. 当前状态不变化
2. 当前回合不推进
3. 界面中显示对应错误提示

---

## 8. AI 模式与自动播放

### 8.1 Baseline

Baseline 采用启发式规则：

1. 优先处理缺粮网络
2. 优先尝试建设高资源潜力城市
3. 在合适时修路合并网络
4. 按固定优先级研发科技
5. 在资源缺口明显时补足对应建筑

### 8.2 Random

Random 每回合从当前合法动作集合中随机选择一个动作。其作用包括：

1. 作为对照组
2. 验证环境闭合性
3. 支撑报告中的基准比较

### 8.3 自动播放展示

1. `Normal`：逐回合播放，适合观察过程
2. `Speed`：一轮刷新推进多回合，适合快速实验

## 9. 界面形式与通用交互规则

### 9.1 技术方案

界面使用：

1. **Python + curses**
2. 鼠标事件捕捉
3. 终端颜色与块字符渲染

### 9.2 鼠标交互规则

1. 点击地图格：选中该格
2. 点击另一格：选中转移
3. 点击当前已选中格：取消选中
4. 点击按钮：执行对应操作
5. 点击记录项：进入详情
6. 滚轮：滚动 Records 列表或详情页

### 9.3 快捷键规则

1. `m`：打开游戏内菜单
2. `b`：优先执行当前可用的建造相关动作
3. `t`：优先执行当前可用的科研相关动作
4. `d`：显示当前选中对象详情
5. `q`：返回、关闭当前层级或回到主菜单
6. 方向键：移动地图选中或滚动记录页面

### 9.4 地图显示规则

1. 每个格子使用块字符表示
2. 城市、道路、地形和选中态都有不同的颜色/纹理
3. 地图区域必须支持鼠标命中
4. 右侧信息区显示分数、回合、资源与当前可执行动作

## 11. Records 与导出字段

### 11.1 Records 目标

Records 不仅用于保存终局结果，也用于支撑后续实验分析：

1. 不同 AI 的最终分数比较
2. 不同 AI 的平均决策速度比较
3. 不同 AI 的整局运行时间比较
4. 不同地图条件下的稳定性分析

### 11.2 记录内容

每条记录至少包含：

1. 基本配置：模式、AI、播放速度、种子、地图大小、地图难度、回合上限
2. 终局结果：分数、城市数、建筑数、科技数、资源结余
3. 动作统计：建城、修路、各类建筑、各类科研、Skip 次数
4. 性能统计：
   - `decision_count`
   - `decision_time_ms_total`
   - `decision_time_ms_avg`
   - `decision_time_ms_max`
   - `turn_elapsed_ms_total`
   - `turn_elapsed_ms_avg`
   - `turn_elapsed_ms_max`
   - `session_elapsed_ms`
5. 终局快照：地图、城市、道路、网络

### 11.3 坐标与 schema

1. 记录中的坐标字段统一使用 `x/y`
2. Records schema 版本独立维护
3. 旧 schema 不做兼容读取

## 15. 方案总结

MicroCiv 的正式方案确定如下：

1. 使用正方形网格地图
2. 城市采集采用八邻域，道路连通采用四邻域
3. 使用 Python + curses 作为正式终端界面方案
4. 界面交互以鼠标为主，并保留高频快捷键
5. 使用 Unicode block elements 完成地图与面状图形绘制
6. 内置 Baseline 与 Random 两种 AI
7. 使用 Records 与 CSV 导出支撑后续实验对比与报告分析
