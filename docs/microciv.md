## 1. 项目定位与阶段目标

### 1.1 项目名称

**MicroCiv：微型文明**

### 1.2 课题定位

本项目对应课程设计第一个选题“微型文明：资源经营、战略规划与智能决策”。项目核心聚焦于“地图—资源—城市—科技—决策”这一闭环，通过规则建模、状态管理、回合推进、经营决策与 AI 自动化，构建一个可运行、可扩展、可比较的微型策略经营环境。

### 1.3 当前阶段

当前主要实现**阶段一：环境搭建与规则驱动的经营决策**，即：

1. 搭建游戏环境与完整规则系统
    
2. 实现地图生成与地图显示
    
3. 实现城市、资源、科技、建筑、回合、评分等核心逻辑
    
4. 实现完整交互流程
    
5. 实现 baseline AI，使其在无人干预下能自动经营文明并稳定输出合法动作
    
6. 为阶段二搜索规划 AI 与 LLM 驱动扩展预留接口与文档结构。
    

### 1.4 阶段一 baseline 目标

阶段一中的 baseline AI 应体现规则性或贪心思想，不能仅使用纯随机策略作为最终版本；其作用包括验证环境正确性、为阶段二提供对照、并形成可运行可比较的基础方案。

### 1.5 阶段二预留方向

阶段二将实现面向未来若干回合的搜索规划智能体，并可进一步探索通过自然语言战略目标驱动搜索权重或规则参数的 LLM 扩展。文档后文已为阶段二预留对应章节。

## 2. 总体设计原则

### 2.1 规则与表现分离

系统采用“**游戏逻辑层**”与“**界面表现层**”解耦设计：

1. 游戏逻辑层负责状态表示、合法性判断、动作执行、资源结算、评分计算、AI 决策等
    
2. 界面表现层负责 TUI 绘制、鼠标/键盘交互、菜单切换、记录展示与导出等
    
3. 阶段二的搜索 AI 与 LLM 扩展直接调用游戏逻辑层接口，不依赖具体 UI。
    

### 2.2 AI 决策与播放速度解耦

Autoplay 模式拆分为两个维度：

1. **AI Policy（决策器类型）**
    
    - Baseline
        
    - Expert
        
    - Custom
        
2. **Playback Mode（播放速度）**
    
    - Normal
        
    - Speed
        

这样可以保证：

- Baseline / Expert / Custom 仅决定“AI 如何决策”
    
- Playback 的 Normal / Speed 仅决定“AI 如何被展示”。
    

### 2.3 地图难度与播放速度解耦

地图生成还包含第三个独立维度：

- **Map Difficulty（地图难度）**
    
    - Normal
        
    - Hard
        

地图难度影响生成参数与资源分布，不影响 AI 决策接口，也不等同于播放速度。所有界面与导出字段中都必须显式区分 `map_difficulty` 与 `playback_mode`，避免“Normal”一词歧义。

### 2.4 阶段一优先级

阶段一重点优先保证：

1. 规则正确
    
2. 状态闭合
    
3. 流程可完整运行
    
4. baseline AI 稳定输出合法动作
    
5. UI 可用且便于调试
    
6. Records 可支撑后续实验对比。

## 3. 地图、坐标与地形设定

### 3.1 地图形状与坐标系

1. 地图采用**对称六边形栅格地图**
    
2. 默认地图规模为参数 `a` 的六边形，其中 `a` 表示中心到任意顶点的距离，包含两端
    
3. 地图大小支持自定义，阶段一正式范围为 `a = 4 ~ 10`
    
4. 地图坐标使用**轴向坐标**
    
5. 六边形地图总格数公式为 `3a(a-1)+1`。
    

### 3.2 地图大小范围

1. 最小地图大小为 `a = 4`，共 37 格，适合快速跑通机制与低深度 AI 调试
    
2. 最大地图大小为 `a = 10`，共 271 格，适合较长对局与大规模经营测试
    
3. 该范围兼顾了机制完整性、TUI 展示可读性与实验规模。
    

### 3.3 地形类型

地图包含以下地形类型：

#### 3.3.1 平原

- 每回合为相邻城市提供 `+3` 粮食
    
- 可建城
    
- 可修路
    
- 颜色：`D9EC37`
    

#### 3.3.2 森林

- 每回合为相邻城市提供 `+2` 木材
    
- 可建城
    
- 可修路
    
- 颜色：`76BB40`
    

#### 3.3.3 山地

- 每回合为相邻城市提供 `+2` 矿石
    
- 可建城
    
- 可修路
    
- 颜色：`C2C2C2`
    

#### 3.3.4 河流

- 每回合为相邻城市提供 `+2` 粮食和 `+1` 科技点
    
- 不可建城
    
- 可修路
    
- 颜色：`52D6FC`
    

#### 3.3.5 荒地

- 不提供资源
    
- 不可建城
    
- 可修路
    
- 颜色：`D19D01`
    

#### 3.3.6 城市

- 不提供资源
    
- 不可修路
    
- 占据 1 格
    
- 颜色：`FF6250`
    

#### 3.3.7 道路

- 不提供资源
    
- 不可建城
    
- 占据 1 格
    
- 颜色：`FFC777`。

## 4. 地图生成规则

### 4.1 初始地图生成约束

游戏开始时，系统自动生成地图，并满足以下条件：

1. 初始地图中不包含城市与道路
    
2. 荒地不会生成在河流相邻格
    
3. 初始地图中必须至少包含 1 个平原格。
    

### 4.2 地形比例与分布规则

为保证地图既具有清晰的区域结构，又保留随机性与多样性，初始地图采用“**区域划分 + 主地形分配 + 区域内混合生成 + 河流叠加 + 规则修补 + 质量检查**”的固定流程。整体算法框架一经确定后不再更换；不同难度仅通过参数调整实现。

### 4.3 生成对象范围

初始地图只生成以下五类基础地形：

1. 平原
    
2. 森林
    
3. 山地
    
4. 河流
    
5. 荒地
    

初始地图中不生成城市与道路。

### 4.4 总体生成流程

地图生成按以下顺序进行：

1. 生成六边形地图全部合法坐标
    
2. 将地图划分为若干连通区域
    
3. 为每个区域分配一种主地形
    
4. 在各区域内部生成以主地形为主、夹杂少量其他地形的混合分布
    
5. 在普通地形上叠加河流
    
6. 执行规则修补
    
7. 执行地图质量检查；若未通过则局部修补或重新生成。
    

### 4.5 地图骨架生成

地图骨架基于轴向坐标生成。系统根据当前地图大小参数，枚举该对称六边形地图中的全部合法格坐标，并以此作为区域划分、邻接判断、河流生成与质量检查的统一基础。

### 4.6 区域划分方法

地图首先被划分为若干个连通区域：

1. 在全部地图格中选取若干个区域种子点
    
2. 区域种子点尽量分散
    
3. 从这些种子点同时向外扩张，将所有地图格划归至某一唯一区域
    
4. 区域划分结果满足：
    
    - 每个格恰好属于一个区域
        
    - 每个区域内部保持连通
        
    - 不同区域构成对整张地图的完整覆盖。
        

### 4.7 主地形分配原则

每个区域指定一种主地形，主地形只从以下四类中选取：

1. 平原
    
2. 森林
    
3. 山地
    
4. 荒地
    

河流不作为区域主地形，而是在后续单独叠加。主地形分配满足：

1. 全图层面存在目标比例控制
    
2. 相邻区域不宜大面积重复同一主地形
    
3. 至少有一个区域以平原为主地形
    
4. Hard 地图可适当降低平原主导区域比例，并提高山地或荒地主导区域比例
    
5. 即使是 Hard 地图，也不得生成明显不可玩的极端结构地图。
    

### 4.8 区域内部地形生成

每个区域的地形生成分为三步：

#### 4.8.1 主地形底填充

区域内全部格子首先统一赋值为该区域主地形。

#### 4.8.2 次级地形斑块注入

在保持区域主地形占优势的前提下，允许混入少量其他普通地形。规则如下：

1. 每个区域可混入若干种次级地形
    
2. 次级地形集合与混入强度由参数控制
    
3. 次级地形以局部斑块形式生成，而非逐格独立随机
    
4. 次级斑块规模较小，不能压过区域主地形总体占比。
    

#### 4.8.3 邻域平滑

区域内部初步生成后，可执行少量邻域平滑，以减少孤立单格、过碎边界与明显噪点，但不得破坏区域主地形的主导地位。

### 4.9 河流叠加规则

普通地形生成完成后，再单独生成河流。河流遵循以下原则：

1. 河流作为特殊结构后叠加
    
2. 每条河流尽量表现为连续路径，而非零散河流格
    
3. 河流通常从地图边缘一侧出发，并延伸至另一侧边缘或其他合理终点
    
4. 河流允许弯曲，但不宜自交、短促折返或无意义绕行
    
5. 河流条数、长度、弯曲程度与分叉情况由参数控制
    
6. Hard 地图可适当提高河流长度、穿越中部概率或其分割感，但仍须保证可玩性。
    

### 4.10 规则修补

#### 4.10.1 荒地与河流邻接修补

若某荒地格与任意河流格相邻，则该荒地格必须被改写为非荒地。改写优先顺序为：

1. 若所属区域主地形不是荒地，则优先改回区域主地形
    
2. 否则优先改为平原
    
3. 若不适合改为平原，则改为森林。
    

#### 4.10.2 平原保底修补

若整张地图中不存在任何平原格，则系统必须强制将至少一个合法格改写为平原。

### 4.11 地图质量检查

规则修补后，还需执行地图质量检查，以避免生成虽合法但明显失衡、过难或缺乏策略意义的地图。检查内容包括但不限于：

1. 可建城地形比例是否过低
    
2. 平原数量是否过少
    
3. 荒地总量是否过高
    
4. 荒地是否形成过大的连续封锁区域
    
5. 河流是否过长或对地图造成过强割裂
    
6. 平原、森林、山地、河流的整体分布是否过度偏科。
    

若质量检查未通过：

- 轻度问题：局部修补
    
- 严重问题：整张地图重新生成。
    

### 4.12 地图难度

#### 4.12.1 Normal 地图

Normal 地图应满足：

1. 平原主导区域数量相对充足
    
2. 荒地比例适中
    
3. 河流主要承担资源与连接价值，不应过度割裂地图
    
4. 各类普通地形分布较为均衡
    
5. 玩家与 baseline AI 都能较稳定找到可行发展路径。
    

#### 4.12.2 Hard 地图

Hard 地图在不改变算法框架的前提下，通过参数调整生成，可包括：

1. 适当减少平原主导区域数量
    
2. 提高山地或荒地主导区域的出现概率
    
3. 增强区域之间的资源偏科程度
    
4. 提高河流长度、弯曲程度或其穿越地图中部的概率
    
5. 适度放宽质量检查中的平衡阈值。
    

但 Hard 地图仍须保证：

1. 不生成死图
    
2. 不生成几乎无合法建城空间的地图
    
3. 不生成极端依赖运气的发展结构
    
4. 仍可用于不同策略 AI 的有效比较。
    

### 4.13 参数化与可复现

为支持实验、Records 重建与不同 AI 策略对比，地图生成至少保存以下参数：

1. 地图大小参数
    
2. 难度档位
    
3. 区域数量或其生成参数
    
4. 普通地形目标比例参数
    
5. 河流生成参数
    
6. 随机种子
    

同一组参数与随机种子应能重现同一张初始地图。

## 5. 资源、科技与建筑系统

### 5.1 资源类型

游戏中包含四种基础资源：

1. 粮食
    
2. 木材
    
3. 矿石
    
4. 科技点。
    

### 5.2 科技树

科技树包含以下四项科技：

1. Agriculture：解锁农场，消耗 6 科技点
    
2. Logging：解锁伐木场，消耗 12 科技点
    
3. Mining：解锁矿场，消耗 20 科技点
    
4. Education：解锁图书馆，消耗 20 科技点。
    

### 5.3 建筑类型

#### 5.3.1 农场

- 消耗 5 木材
    
- 提供 +3 粮食/回合
    
- 需要 Agriculture
    

#### 5.3.2 伐木场

- 消耗 5 木材
    
- 提供 +2 木材/回合
    
- 需要 Logging
    

#### 5.3.3 矿场

- 消耗 6 木材 + 4 矿石
    
- 提供 +2 矿石/回合
    
- 需要 Mining
    

#### 5.3.4 图书馆

- 消耗 6 木材 + 4 矿石
    
- 提供 +2 科技点/回合
    
- 需要 Education。
    

### 5.4 建筑上限

1. 每座城市至多建造 6 座建筑
    
2. 同种建筑可以重复建造
    
3. 建筑归属于单个城市，不属于整个城市网络。
    

### 5.5 资源图标与建筑映射

界面中四种资源与建筑固定映射如下：

1. 粮食 → 农场
    
2. 木材 → 伐木场
    
3. 矿石 → 矿场
    
4. 科技点 → 图书馆。

## 6. 城市、道路与城市网络

### 6.1 建城条件

新城市可以建立在任意合法格上，不要求与已有城市、道路或河流相邻。

合法建城格定义为：

1. 当前不是城市
    
2. 当前不是道路
    
3. 当前地形允许建城，即不是河流、荒地。
    

### 6.2 修路条件

新道路必须建立在合法修路格上，且必须与至少一个已有城市或道路相邻。

合法修路格定义为：

1. 当前不是城市
    
2. 当前不是道路
    
3. 当前地形允许修路
    
4. 与至少一个已有城市或道路相邻。
    

### 6.3 城市网络定义

若两个城市之间存在一条仅由**城市、道路、河流**组成的连通路径，则这两个城市属于同一城市网络。说明如下：

1. 城市是城市网络中的合法连接节点
    
2. 道路与河流都可作为网络连接通路的一部分
    
3. 城市本身不是道路，城市格上仍不可修路
    
4. 一座城市若同时与多座城市连通，则它们天然属于同一连通网络。
    

### 6.4 城市网络共享内容

同一城市网络共享：

1. 粮食
    
2. 木材
    
3. 矿石
    
4. 科技点
    
5. 已解锁科技。
    

不同城市网络之间，上述内容彼此独立。

### 6.5 网络合并

若一次成功建城或修路使多个原本独立的城市网络连通，则在该动作执行后立刻合并网络：

1. 四类资源数量相加
    
2. 已解锁科技取并集
    
3. 城市统一归入同一网络
    
4. 合并后再进入资源结算阶段。

## 7. 资源产出与归属规则

### 7.1 城市基础资源范围

每座城市根据其周围 1 格范围内的区域计算基础资源产出，不包含城市自身所在格。

### 7.2 资源格只计算一次

若一个资源格与多座城市相邻，则该格资源只计算一次。

### 7.3 资源归属判定

当一个资源格与多座城市相邻时，按该资源对应建筑数量最多的城市确定归属：

1. 平原、河流中的粮食部分比较农场数量
    
2. 森林比较伐木场数量
    
3. 山地比较矿场数量
    
4. 河流中的科技点部分比较图书馆数量。
    

### 7.4 并列时的判定

若对应建筑数量并列，则按以下顺序决定归属：

1. 建筑数量更多者优先
    
2. 建城时间更早者优先
    
3. 坐标字典序更小者优先。
    

### 7.5 归属刷新时机

每次成功执行动作后，在当回合资源结算前统一重新计算全部资源格归属。

## 8. 特殊建造与覆盖规则

### 8.1 覆盖地形

建城或修路会覆盖格子原本的地形类型。

### 8.2 一次性获得原地形资源

若原地形本来提供资源，则在建造完成时立刻一次性获得对应资源。

阶段一统一设定为：

- 一次性获得 **8 单位**对应资源。
    

### 8.3 河流上修路的额外消耗

若在河流格上修路，需要额外满足以下二选一：

1. 消耗 `x` 单位木材
    
2. 或消耗 `y` 单位矿石
    

支付顺序固定为：

1. 若木材足够，则优先消耗木材
    
2. 否则若矿石足够，则消耗矿石
    
3. 否则修路失败。
    

### 8.4 河流修路的扣费来源

修路所需额外资源从与该道路相邻并使其合法的已有城市网络中扣除。

若待修道路格同时与多个独立城市网络相邻，则按系统固定顺序自动选择一个满足支付条件的网络扣费；若没有任何相邻网络满足条件，则修路失败。

## 9. 初始状态

1. 游戏开始时地图上没有城市，因此不存在城市网络
    
2. 初始没有共享资源池
    
3. 初始没有已解锁科技
    
4. 第一座城市建立后，形成第一个城市网络，并开始累积资源与科技状态。

## 10. 回合与结算系统

### 10.1 总回合数

阶段一总回合数支持自定义，正式范围为：

- 最小值：30 回合
    
- 最大值：150 回合。
    

### 10.2 每回合基本流程

每回合按如下顺序进行：

1. 玩家或 AI 选择一个动作
    
2. 系统校验动作是否合法、资源是否足够
    
3. 若动作合法且执行成功，则：
    
    1. 执行动作
        
    2. 若导致网络合并，则立刻合并城市网络
        
    3. 重新计算资源格归属
        
    4. 所有城市结算本回合资源产出
        
    5. 结算粮食消耗
        
    6. 刷新当前分数
        
    7. 若当前回合未达到总回合数，则回合数加一
        
    8. 若当前回合已达到总回合数，则直接进入终局界面
        
4. 若动作不合法或执行失败，则：
    
    - 不执行动作
        
    - 不结算资源
        
    - 不结束回合
        
    - 不增加回合数
        
    - 保留当前操作状态并显示失败提示。
        

### 10.3 粮食消耗规则

1. 每回合结算时，每座城市的粮食减少 **4** 单位
    
2. 先结算资源产出，再结算粮食减少
    
3. 若某城市在本回合结算开始前，其所属网络中的粮食数量小于等于 0，则该城市在本回合中除粮食外的其他资源产出减少 **100%**，即停工。
    

### 10.4 设计意图

阶段一数值的核心意图是通过高粮食消耗与停工惩罚，强化“资源错位 + 路网调度”的玩法。纯山地或纯森林附近的孤立城市若不及时连入农业网络，极易断粮停工，因此道路与网络连通是核心战略内容。

### 10.5 成功动作才结束回合

以下动作在**成功执行**后结束回合：

1. 成功建城
    
2. 成功修路
    
3. 成功建造建筑
    
4. 成功研究科技
    
5. Skip。
    

以下情况不会结束回合：

1. 资源不足
    
2. 科技未解锁
    
3. 科技点不足
    
4. 建筑数量达到上限
    
5. 非法格建城或修路
    
6. 点击 Cancel
    
7. 点击已研究科技
    
8. 非法点击或无效按键。

## 12. 评分系统

### 12.1 终局目标

在限定回合数内最大化文明总分。

### 12.2 分数组成

总分由以下部分组成：

1. 城市数量
    
2. 已建成建筑数量
    
3. 已解锁科技数量
    
4. 四种资源各自总剩余数量。
    

### 12.3 评分函数

阶段一采用如下加权线性评分公式：

[  
S = W_c \cdot N_{city} + W_b \cdot N_{bldg} + W_t \cdot N_{tech} + \lfloor \frac{R_{food}}{K_f} \rfloor + \lfloor \frac{R_{wood}}{K_w} \rfloor + \lfloor \frac{R_{ore}}{K_o} \rfloor + \lfloor \frac{R_{sci}}{K_s} \rfloor  
]

其中参数为：

- 城市分 (W_c = 20)
    
- 建筑分 (W_b = 15)
    
- 科技分 (W_t = 30)
    
- 粮食折算 (K_f = 10)
    
- 木材折算 (K_w = 5)
    
- 矿石折算 (K_o = 5)
    
- 科技点折算 (K_s = 20)。
    

### 12.4 评分逻辑说明

1. 鼓励把资源转化为城市、建筑与科技，而不是单纯囤积
    
2. 防止通过终局前临时破坏资源格来刷一次性资源分
    
3. 科技点采用较高折算系数，避免后期纯堆图书馆导致失衡。
    

### 12.5 当前分数

1. 当前分数定义为按终局评分函数实时计算得到的暂时分数
    
2. 每次成功动作完成并完成当回合结算后刷新
    
3. 失败操作、取消操作、选中切换时不刷新
    
4. 进入终局界面前再刷新一次。
    

### 12.6 终局触发

设总回合数为 `T`，当前回合数初始为 `1`。

- 当第 `r` 回合成功执行动作并完成当回合结算后：
    
    - 若 `r < T`，进入第 `r+1` 回合
        
    - 若 `r = T`，直接进入终局界面
        

不会显示第 `T+1` 回合。

## 22. Autoplay 模式

### 22.1 进入方式

从初始菜单选择 Autoplay，并在地图选择界面点击 Start 后进入。

### 22.2 Playback: Normal

在 Playback 的 Normal 模式下：

1. AI 以固定节奏逐回合执行动作
    
2. 常规情况下，每隔约零点几秒执行一回合
    
3. 若某回合决策耗时短于显示间隔，则达到显示间隔后再进入下一回合
    
4. 若某回合决策耗时长于显示间隔，则在决策完成后立即进入下一回合。
    

### 22.3 Playback: Speed

在 Playback 的 Speed 模式下：

1. AI 以尽可能快的速度连续执行回合
    
2. 主要用于快速得到终局结果
    
3. 界面不按每回合刷新，而采用低频批量刷新策略，以减少 TUI 重绘开销。
    

#### 22.3.1 基本原则

Speed 模式将“逻辑执行速度”与“界面刷新频率”分离处理：

1. AI 决策、动作执行、资源结算、回合推进等逻辑部分尽可能连续运行
    
2. 地图、分数、回合数、提示信息等界面部分不要求在每一回合后立即重绘
    
3. 通过降低刷新频率，提高整体自动运行速度。
    

#### 22.3.2 刷新触发条件

满足以下任一条件时，立即进行一次刷新：

1. 自上次刷新以来，已经过固定时间间隔
    
2. 自上次刷新以来，已累计执行若干个成功回合
    
3. 当前对局已进入终局
    
4. 用户触发需要立即响应的输入或界面切换操作。
    

默认实现建议为：

- 约 100ms 或约 10 回合刷新一次，以先到者为准。
    

#### 22.3.3 刷新时显示内容

每次刷新时，界面至少应保证以下内容正确显示：

1. 当前地图状态
    
2. 当前分数
    
3. 当前回合数
    
4. 当前模式与 AI 类型
    
5. 必要的运行提示信息。
    

可选地显示最近一步动作摘要，但不是阶段一强制要求。

#### 22.3.4 不刷新的情况

在两次刷新之间，系统可以连续执行多个回合而不立即重绘。此时：

1. 游戏内部状态应正常推进
    
2. AI 决策与资源结算应完整执行
    
3. 不要求用户在每一步都看到中间过程
    
4. 只要达到下一次刷新条件，界面即更新为当前最新状态。
    

#### 22.3.5 终局时的强制刷新

无论是否达到时间间隔或回合阈值，只要对局结束，系统都必须立即执行一次完整刷新，并随后进入终局界面。

#### 22.3.6 与 Normal 的区别

- Playback 的 Normal 强调逐回合展示 AI 行为，便于观察过程
    
- Playback 的 Speed 强调快速完成对局，便于批量实验与结果比较。
    

#### 22.3.7 对交互的要求

在 Speed 模式下，玩家仍不能手动参与建造或科研，但应继续支持：

1. 按 `m` 打开菜单
    
2. 按 `q` 直接退出。
    

当用户触发上述操作时，系统应先处理必要刷新，再切换到对应界面或执行退出逻辑，以避免显示状态与实际状态不一致。

### 22.4 玩家在 Autoplay 中可做的事

1. 不能手动点击地图进行建造或科研
    
2. 仍可按 `m` 打开菜单
    
3. 仍可按 `q` 直接退出。

1. 终局地图状态
    
2. 城市列表与坐标

4. 各城市的建筑分布
    
5. 各城市网络的资源与科技状态
    
6. 模式、AI 类型、Custom 输入内容
    
7. 地图难度、地图参数、随机种子
    
8. 最终分数与八项统计数据
    
9. 对局时间、版本、参数配置等元信息。
    

#### 24.4.3 两种格式的分工

- JSON 用于“状态结构层面”的完整保存与终局重建。
    

#### 24.4.4 单一格式实现时的优先级

若阶段一出于实现成本考虑只实现一种导出格式，则优先实现 **CSV**。JSON 可作为后续扩展接口预留。

## 26. 阶段一：规则驱动经营 AI（Baseline）

### 26.1 阶段目标

阶段一需要完成一个 baseline 智能体，使其在无人干预下自动经营文明，并体现基本经营逻辑，能够稳定优于随机策略。该目标与原始课程设计要求一致。

### 26.2 baseline AI 的最低要求

1. 在任何合法状态下都能输出一个合法动作
    
2. 不能只依赖纯随机
    
3. 应体现规则性或贪心思想
    
4. 可在实验中与随机策略作对比。
    

### 26.3 baseline AI 建议能力

baseline 至少应具备以下能力：

1. 根据资源情况选择优先发展方向
    
2. 在多个合法建城位置中选择相对更优的位置
    
3. 在已有城市中选择较合理的建筑顺序
    
4. 在多个科技中选择下一项研究目标。
    

### 26.4 baseline 决策框架建议

可采用如下总体优先级：

> 资源补短板 > 高收益建城 > 关键科技 > 其他建筑 > 跳过。

#### 26.4.1 建城决策

1. 枚举所有合法建城位置
    
2. 为每个位置计算简单评分
    
3. 评分可考虑周围若干格可提供的粮食、木材、矿石、科技点
    
4. 优先选择综合资源潜力更高的位置。
    

#### 26.4.2 建筑决策

当某类资源产出明显不足时，优先建造对应建筑，例如：

- 粮食偏低 → 农场
    
- 木材偏低 → 伐木场
    
- 科技点偏低 → 图书馆。
    

#### 26.4.3 科技决策

若某类建筑尚未解锁，且文明发展明显受限于该类资源，则优先研究对应科技，例如：

- 缺乏粮食增产手段 → 优先 Agriculture
    
- 缺乏科技点来源 → 优先 Education。

## 28. 阶段二：LLM 驱动的战略目标转化（Custom）

### 28.1 方向说明

阶段二可选探索方向之一是：支持自然语言形式的战略目标，并使用大语言模型将自然语言目标转化为评分函数权重或规则参数，再与搜索规划模块结合。

### 28.2 Custom AI 的定位

Custom 模式不是独立替代搜索器的纯 LLM 落子器，而是：

- **上层**：由自然语言战略目标生成偏好、权重或规则参数
    
- **下层**：由搜索规划或规则引擎执行具体动作选择。
    

即采用“高层目标解释 + 底层精确决策”的结构。

### 28.3 预留接口

预留接口包括：

1. 自然语言战略目标输入
    
2. LLM 输出的权重参数
    
3. 权重到评估函数/动作优先级的映射
    
4. 参数记录与回放
    
5. 与 baseline / Expert 的实验对比。
    

### 28.4 示例目标

预留支持的自然语言目标示例：

- 优先科技发展，谨慎扩张
    
- 前期优先保证粮食和木材，后期冲高分
    
- 偏向多城市扩张
    
- 保守经营，避免资源断档。

## 29. TUI 技术方案

### 29.1 设计目标

阶段一的 TUI 不仅用于显示普通文本，还需要承担以下任务：

1. 展示地图、菜单、城市面板、Records、终局界面等完整界面结构
    
2. 支持鼠标点击、键盘快捷键、滚动浏览、页面切换与 Autoplay 动态刷新
    
3. 支持具有一定视觉设计感的标题字、分数字、LOGO 与核心英文文本
    
4. 为阶段二继续扩展搜索 AI、LLM 驱动模式与实验记录界面保留良好的可维护性
    

因此，本项目的 TUI 方案不采用简单的逐行 `print()` 输出，而采用“**TUI 框架 + 自定义字形渲染**”的组合方案。

### 29.2 总体技术路线

TUI 层采用如下总体方案：

1. **主框架：Textual**
    
2. **富文本与颜色能力：Rich 生态**
    
3. **视觉文本层：自定义点阵字模渲染**
    
4. **LOGO：自定义六边形图标组合渲染**
    
5. **`pyfiglet`：仅用于开发早期占位或参考，不作为最终正式视觉方案**
    

本项目不使用 `durdraw`、`bit` 等工具作为主实现路径。

### 29.3 选择 Textual 作为主框架的原因

Textual 作为 Python 生态中较成熟的现代 TUI 框架，适合本项目原因如下：

1. 支持较复杂的界面布局，适合实现“左侧地图区 + 右侧信息区 + 底部提示区”等结构
    
2. 支持鼠标事件，便于实现地图点击、菜单点击与 Records 选择
    
3. 支持键盘事件，便于实现 `m`、`b`、`q`、`d`、`t` 等快捷键
    
4. 支持滚动区域，适合实现 Records 列表与详细记录页面
    
5. 支持自定义 Widget，便于将地图、LOGO、分数字、像素风标题等封装为独立组件
    
6. 有利于将界面结构、状态更新与业务逻辑解耦，便于后续维护与扩展
    

### 29.4 不采用其他工具作为主方案的原因

#### 29.4.1 pyfiglet

`pyfiglet` 适合快速生成标题原型或作为开发早期占位工具，但不适合作为本项目正式视觉方案，原因包括：

1. 难以精细控制字体外形与整体风格
    
2. 不便于实现统一的像素游戏风设计
    
3. 不便于实现有明确阴影、轮廓与配色控制的大数字和标题字
    
4. 不利于后续与游戏 UI 风格保持统一
    

因此，`pyfiglet` 仅作为开发过程中的临时参考工具，不作为正式字体渲染方案。

#### 29.4.2 durdraw 与 bit

`durdraw`、`bit` 等工具更适合做独立的 ASCII/ANSI 图形草图、手工字符画或简单动画，不适合作为本项目主方案，原因包括：

1. 不适合构建完整的 Python TUI 应用主框架
    
2. 与项目业务逻辑、状态切换、交互系统的整合价值较低
    
3. 在后期维护、扩展与动态界面更新方面不如 Textual 方案清晰
    
4. 不适合承载地图区、菜单区、Records、Autoplay 刷新等完整应用界面需求
    

因此，这类工具不作为正式开发的主技术路线。

### 29.5 视觉文本层方案：自定义点阵字模

为实现具有设计感的数字与英文文本，本项目采用“**自定义点阵字模 + 终端彩色块渲染**”的方案，而不是依赖终端默认字体或通用 ASCII 大字库。

该方案的核心思想为：

1. 将数字、字母和常用符号定义为若干套点阵模板
    
2. 使用彩色块字符或彩色背景块在终端中绘制这些模板
    
3. 通过渲染偏移实现阴影效果
    
4. 通过不同字号模板支持标题、选项字、分数字等不同用途
    

### 29.6 点阵字模的适用范围

自定义点阵字模优先用于以下内容：

1. `MicroCiv` 标题
    
2. `SCORE` 与当前分数字
    
3. `STEP` 与当前回合数字
    
4. 主菜单、地图选择界面、终局界面中的核心英文标题
    
5. 其他需要明显视觉强调的短文本
    

普通说明性文本不强制使用点阵字模，仍可采用标准终端字体，例如：

1. 错误提示文本
    
2. 调试信息
    
3. 导出状态
    
4. Records 中较长的说明类文本
    
5. 其他非重点装饰文本
    

这样可以在控制工作量的同时，突出关键界面的视觉层次。

### 29.7 点阵字模的建议分层

建议至少维护以下几类字模：

1. **大号数字字模**
    
    - 用于显示当前分数与回合数中的数字
        
    - 支持阴影渲染
        
2. **中号标题字模**
    
    - 用于 `MicroCiv`、`SCORE`、`STEP` 等标题词
        
    - 风格统一，便于构建游戏感
        
3. **小号像素字模**
    
    - 用于主菜单与部分按钮标签
        
    - 兼顾可读性与风格统一
        

字模可用以下任一结构定义：

1. 字符串数组
    
2. 0/1 矩阵
    
3. 坐标点集
    

具体存储方式由实现决定，但应便于：

- 控制颜色
    
- 控制字距
    
- 控制行距
    
- 叠加阴影
    
- 后续扩展字符集
    

### 29.8 阴影效果实现方案

为了实现类似像素游戏中带立体感的大数字与标题字，字形渲染应支持阴影层。

建议实现方式如下：

1. 先绘制阴影层，位置相对主字形偏移一个固定量
    
2. 再绘制主字形
    
3. 阴影颜色使用更暗的同色系或统一阴影色
    
4. 阴影偏移量和颜色可配置
    

这样可以在终端中实现带阴影的大数字与标题字，例如当前分数字、回合数字等。

### 29.9 终端中的字符绘制建议

点阵字模与 LOGO 的渲染建议使用以下方式之一：

1. 彩色背景空格块
    
2. `█` 等实心块字符
    
3. `▀`、`▄` 等半块字符，用于提高垂直方向精细度
    

推荐优先使用“彩色背景空格块”或“实心块字符”模拟像素风块面，以获得更稳定的视觉效果。具体选择可依据实际终端兼容性决定。

### 29.10 LOGO 方案

LOGO 采用七种地块颜色的六边形图标组合，构成固定的图形标识。LOGO 不依赖外部工具自动生成，而作为项目内部的自定义渲染组件实现。

LOGO 方案建议如下：

1. 将每个六边形图块视为固定模板
    
2. 为每个图块设置相对坐标与颜色
    
3. 使用块字符或背景色在终端中绘制六边形近似图形
    
4. 七个图块按既定布局组合成完整 LOGO
    
5. LOGO 可在初始菜单、游戏内菜单、终局界面、Records 等页面复用
    

由于终端字符并非真正正方像素，LOGO 不强求几何上完全精确的六边形轮廓，而以“整体视觉上清晰可辨”为优先目标。

### 29.11 TUI 中的模块划分建议

TUI 层建议至少划分为以下模块：

1. **主应用模块**
    
    - 管理页面切换与全局事件
        
2. **地图渲染模块**
    
    - 负责绘制六边形地图、格子高亮、地形颜色等
        
3. **右侧面板模块**
    
    - 负责显示分数、回合、资源、科技、菜单项与提示信息
        
4. **LOGO 渲染模块**
    
    - 负责绘制项目 LOGO
        
5. **字体渲染模块**
    
    - 负责点阵字模与阴影文字渲染
        
6. **Records 视图模块**
    
    - 负责滚动、选择、跳转顶部/底部与详细界面展示
        
7. **Autoplay 展示控制模块**
    
    - 负责 Normal / Speed 模式下的刷新协调
        

上述模块应尽量与游戏逻辑层解耦，避免将规则判断与界面绘制强绑定。

### 29.12 与游戏逻辑层的关系

TUI 层只负责表现与交互，不直接持有复杂规则逻辑。推荐交互方式如下：

1. 游戏逻辑层提供标准状态接口
    
2. TUI 层读取当前状态并进行绘制
    
3. 鼠标点击与键盘输入先转化为抽象动作或事件
    
4. 由游戏逻辑层判定是否合法并更新状态
    
5. TUI 再根据最新状态重绘界面
    

这样可保证：

- 阶段一 baseline AI 能直接复用逻辑层
    
- 阶段二搜索规划 AI 不受 UI 限制
    
- Custom / LLM 模块也能直接接入逻辑层
    
- 后续若更换 GUI 方案，不需要重写核心规则
    

### 29.13 开发阶段的实现优先级建议

TUI 部分建议按以下顺序推进：

#### 29.13.1 第一阶段：主框架可运行

优先实现：

1. Textual 主应用骨架
    
2. 页面切换
    
3. 地图区与右侧面板区布局
    
4. 鼠标与键盘事件
    
5. 基本文本显示与状态刷新
    

#### 29.13.2 第二阶段：核心视觉组件

在主框架稳定后，再实现：

1. LOGO 渲染组件
    
2. 大数字字模
    
3. 标题字模
    
4. 基础阴影效果
    

#### 29.13.3 第三阶段：风格统一与细节优化

最后再逐步完善：

1. 菜单项像素风文本
    
2. 分页与滚动细节
    
3. Autoplay 下的视觉反馈
    
4. Records 页的美化
    
5. 终局界面的统一风格
    

### 29.14 阶段一的视觉重点建议

为控制开发复杂度，阶段一优先美术化以下内容：

1. `MicroCiv` 标题
    
2. `SCORE` 与分数字
    
3. `STEP` 与回合数字
    
4. LOGO
    
5. 主菜单和地图选择界面的核心英文标题
    

其余说明性文本可暂时保持终端普通字体，以保证开发效率与可维护性。

### 29.15 方案总结

阶段一 TUI 的正式技术方案确定如下：

1. 使用 **Python + Textual** 作为主 TUI 框架
    
2. 使用 **Rich 生态** 提供底层颜色与文本能力
    
3. 采用**自定义点阵字模渲染**实现标题字、数字字与像素风英文文本
    
4. 采用**自定义 LOGO 渲染组件**实现七地块六边形组合图标
    
5. `pyfiglet` 仅用于开发早期占位或参考，不作为最终视觉方案
    
6. `durdraw`、`bit` 等工具不作为主实现路径
    
7. TUI 层与游戏逻辑层保持解耦，便于阶段二继续扩展
