# MicroCiv TUI 完整实现流程与规范

本文档以 [docs/microciv.md](/home/zolive/microciv/docs/microciv.md) 第 `13` 到 `24` 节及第 `29` 节为唯一上位依据，把阶段一 TUI 的页面流程、布局、交互和视觉方案整理成可直接照着实现的完整规范。

目标不是“再做一个简化版 TUI”，而是把 `microciv.md` 中已经写明的界面要求真正落成。

## 1. 总原则

1. TUI 必须全屏利用终端空间，不允许把主界面只缩在屏幕中央一小块。
2. 鼠标是主要操作方式，键盘是补充快捷键。
3. 逻辑层与 TUI 层严格分离。
4. 页面切换、选中状态、提示信息、Autoplay 刷新都必须遵守原始文档，而不是由当前实现自由发挥。
5. 阶段一可以先不把所有视觉细节做到最终极，但布局、流程和信息层级必须先对。
6. `md` 里的纯字符示意图只用于表达结构与比例关系；实际程序必须比文档更注重对齐，不能把示意图里的字符误差带进正式界面。

## 2. 统一交互规则

依据 `13.1` 到 `13.4`，全局规则固定如下。

### 2.1 输入方式

1. 主要通过鼠标点击。
2. 允许键盘快捷键辅助操作。
3. 纵向选项默认居中对齐。
4. 地图格本体只使用六边形外形与颜色表达信息；数字和文字只出现在右侧面板、标题字或说明区中。

### 2.2 选中规则

1. 同一时刻最多只有一个地图格可处于选中状态。
2. 点击未选中的格：
   该格变为选中，并出现白色选中框。
3. 点击另一个格：
   选中状态转移。
4. 点击当前已选中的格：
   取消选中。
5. 点击 `Cancel`：
   取消当前选中状态。
6. 从菜单返回游戏后：
   不能保留进入菜单前的选中状态。

### 2.3 提示信息区

1. 游戏主界面右侧底部必须保留一块小字信息区。
2. 手动模式下，该区域用于显示失败原因或必要提示；无提示时可留空。
3. Autoplay 模式下，该区域第一行显示 `mode / ai`，出现提示时占第二行。
4. 提示区只属于右侧信息栏，左侧地图底部不留任何额外提示空间。
5. 提示在以下情况清除：
   - 下一次点击
   - 成功执行动作
   - 页面切换

### 2.4 快捷键

1. `q`
   任何界面直接退出程序。
2. `m`
   - 游戏主界面：打开游戏内菜单
   - 游戏内菜单：等同 `Continue`
3. `b`
   - 仅 Records 相关界面有效
   - 返回上一级界面
4. `d`
   - Records 可滚动界面跳到底部
5. `t`
   - Records 可滚动界面跳到顶部
6. 鼠标滚轮只用于 Records 界面。
7. 游戏主界面、地图选择界面、终局界面都不应依赖滚轮完成正常操作。

## 3. 页面总流程

完整页面状态机必须是下面这条链，而不是自由拼接：

```text
+------------------+
|   Initial Menu   |
| Play / Autoplay  |
| Records / Exit   |
+---------+--------+
          |
          | Play
          v
+------------------+
|   Map Select     |
|   mode = Play    |
+---------+--------+
          |
          | Start
          v
+------------------+        m        +------------------+
|   Game Screen    | --------------> |   In-game Menu   |
|   manual play    | <-------------- | Continue/Menu/...|
+---------+--------+       m         +------------------+
          |
          | final turn
          v
+------------------+
|   Final Screen   |
| Restart/Menu/Exit|
+---------+--------+
          |
          | Restart
          v
+------------------+
|   Map Select     |
+------------------+
```

Autoplay 与 Play 的区别只在模式和地图选择项，不在总流程上另起一套界面结构：

```text
Initial Menu
   |
   +--> Play ------> Map Select (Play) ------> Game Screen (Human)
   |
   +--> Autoplay --> Map Select (Autoplay) --> Game Screen (AI Auto)
   |
   +--> Records ---> Record List ---> Record Detail
   |
   +--> Exit
```

## 4. 初始菜单界面

依据第 `14` 节，初始菜单必须严格如下。

### 4.1 结构

```text
+--------------------------------------------------------------------------+
|                                                                          |
|  [ 左侧：LOGO + MicroCiv ]                [ 右侧：纵向四项菜单 ]             |
|                                                                          |
|                   __                             Play                     |
|                __/  \__                         Autoplay                 |
|               /  \__/  \                        Records                  |
|               \__/  \__/                        Exit                     |
|               /  \__/  \                                                 |
|               \__/  \__/                                                 |
|                  \__/                                                    |
|                                                                          |
+--------------------------------------------------------------------------+
```

### 4.2 必须满足

1. 左侧只放 LOGO 与游戏名 `MicroCiv`。
2. 右侧只放四个选项：
   - `Play`
   - `Autoplay`
   - `Records`
   - `Exit`
3. 此界面不应该提前出现：
   - `Baseline`
   - `Normal`
   - `Speed`
   - 地图参数描述
   - AI 类型描述
4. `Play` 和 `Autoplay` 本身只是入口，不应承载地图配置细节。

### 4.3 点击后动作

1. 点击 `Play`
   进入地图选择界面，模式为手动。
2. 点击 `Autoplay`
   进入地图选择界面，模式为自动。
3. 点击 `Records`
   进入历史记录列表。
4. 点击 `Exit`
   退出程序。

依据第 `15` 节，Play 和 Autoplay 共用同一类地图选择界面。

### 5.1 总布局

```text
|  +---------------------------+   +------------------------------------+  |
|  |                           |   |                                    |  |

|  |   / \ / \ / \ / \         |   |  [Map Size]                        |  |
|  |   \_/ \_/ \_/ \_/         |   |  [Turn Limit]                      |  |
|  |                           |   |                                    |  |
|  |                           |   |  Autoplay 时追加：                  |  |
|  |                           |   |  [AI Type]                         |  |
|  |                           |   |  [Custom Input] (仅 Custom 显示)    |  |
|  |                           |   |  [Playback]                        |  |

|                                  |  [Menu]                            |  |
|                                  +------------------------------------+  |
+--------------------------------------------------------------------------+
```

### 5.2 公共项

Play 和 Autoplay 都必须有：

1. `Map Difficulty`
   - `Normal`
   - `Hard`
2. `Map Size`
   - 阶段一范围 `4 ~ 10`
3. `Turn Limit`
   - 阶段一范围 `30 ~ 150`
4. `Start`
5. `Recreate`
6. `Menu`

### 5.3 Recreate

1. 重新生成地图。
2. 保留当前：
   - `Map Difficulty`
   - `Map Size`
   - 其他地图配置
3. 不能把配置偷重置为默认值。

### 5.4 Menu

1. 返回初始菜单。
2. 不进入游戏。

### 5.5 Start

1. 用当前模式和当前配置进入游戏主界面。

### 5.6 Autoplay 专属项

只有从初始菜单点 `Autoplay` 进入地图选择界面时，才应额外出现：

1. `AI Type`
   - `Baseline`
   - `Expert`
   - `Custom`
2. `Custom` 输入栏
   - 仅在 `AI Type = Custom` 时显示
3. `Playback`
   - `Normal`
   - `Speed`

所以 `Baseline`、`Normal`、`Speed` 的正确位置都在这里，而不是初始菜单。

## 6. 游戏主界面

依据第 `16` 节，主界面默认信息非常克制，不能把整块右侧长期堆成大滚动面板。

### 6.1 默认布局

```text
+--------------------------------------------------------------------------+
|  +-------------------------------------------+   +--------------------+  |
|  |                                           |   | SCORE              |  |
|  |                                           |   | 0000               |  |
|  |             左侧：正方形地图区               |   |                    |  |
|  |         上下几乎占满屏幕高度                 |   | STEP               |  |
|  |         仅与屏幕边缘留小空隙                 |   | 01                 |  |
|  |                                           |   |                    |  |
|  |      __    __    __    __                |   | Skip               |  |
|  |     /  \__/  \__/  \__/  \               |   |                    |  |
|  |     \__/  \__/  \__/  \__/               |   | ------------------ |  |
|  |     /  \__/  \__/  \__/  \               |   | 小字信息区           |  |
|  |                                           |   | tip / status       |  |
|  |                                           |   +--------------------+  |
|  +-------------------------------------------+                           |
+--------------------------------------------------------------------------+
```

### 6.2 默认只显示

1. 左侧地图。
2. 右侧上方当前分数。
3. 右侧下方当前回合数。
4. 右侧 `Skip`。
5. 右侧底部小字信息区。
6. 左侧地图区本身从上到下铺开，不为提示区让位。

### 6.3 不应默认常驻

1. 大块资源状态表。
2. 大块科技状态表。
3. 所有建造按钮一次性全列出。
4. 需要靠滚动才能看到完整操作项的长操作栏。
5. 任何必须用滚轮才能操作的游戏内面板。

这些内容只应在对应上下文中出现：

1. 点击普通格进入建造面板时出现建造选项。
2. 点击城市时出现城市操作面板。
3. 点击科技或资源图标时再进入更深一级确认面板。

### 6.4 游戏内菜单

按 `m` 后，主界面消失，切到游戏内菜单：

```text
+--------------------------------------------------------------------------+
|                                                                          |
|  [ 左侧：LOGO + MicroCiv ]                 [ 右侧：Continue ]              |
|                                           [      Menu     ]              |
|                                           [      Exit     ]              |
|                                                                          |
+--------------------------------------------------------------------------+
```

1. `Continue`
   返回游戏。
2. `Menu`
   返回初始菜单，不保存当前进度。
3. `Exit`
   退出程序。
4. 再按一次 `m`
   等同 `Continue`。

## 7. 地图区的正式要求

地图区必须是界面主角，面积优先级高于右侧面板。

### 7.1 全屏利用原则

1. 一个对称六边形地图可以被包含在一个正方形区域中。
2. 左侧地图区应当就是这个近乎铺满高度的正方形区域。
3. 该正方形区域上下几乎占满屏幕，只与屏幕边缘保留小空隙。
4. 右侧面板应为窄栏，不允许把地图压成中央小窗。
5. 终端全屏时，地图正方形区域和右侧信息栏都应同步扩张。

### 7.2 六边形格子

地图不能只用紧贴字符按钮草率代替六边形概念，正式方案应使用“六边形近似字符模板”，并保证六边形之间整齐排列。

建议每格至少使用 3 行结构，例如：

```text
  __
 /  \
 \__/
```

不要使用下面这种尖顶写法代替六边形：

```text
 /\ 
/..\ 
\__/ 
```

### 7.3 颜色表达

每个格要能看出：

1. 地形颜色
2. 城市 / 道路覆盖状态也必须通过颜色体系表达
3. 是否被选中

进一步固定为：

1. 地图格、LOGO 六边形、资源信息区六边形图标默认都只有颜色，没有边框。
2. 地图上只有“当前被选中的格子”会额外出现白色边框。
3. 不能把默认黑边、灰边或白边当作常规视觉元素。

### 7.4 允许的地图层次

每个格固定为：

1. 六边形外形
2. 六边形内部的纯颜色填充

进一步约束如下：

1. 格内不放文字。
2. 格内不放数字。
3. 格内不放城市或道路图标。
4. 格内不放覆盖物字符。
5. 城市、道路与普通地形的差异都只通过颜色表达。

### 8.1 既不能建城也不能修路

### 8.2 可建城或修路

右侧切到建造面板。

#### 8.2.1 既可建城又可修路

```text
+------------------------------+
| [ City ]   [ Road ]          |
|   ^白框默认在 City             |
|                              |
| Build                        |
| Cancel                       |
| ---------------------------  |
```

规则：

2. 默认城市图标有白框。
3. 点击图标可切换当前建造类型。

#### 8.2.2 仅可建城

```text
+------------------------------+
```

### 8.3 Build / Cancel 的界面规则

1. `Build` 成功

   - 结算
   - 刷新分数
   - 进下一回合或进终局
2. `Build` 失败
   - 不结束回合
   - 保留当前面板
   - 在右下小字信息区给出失败原因
3. `Cancel`
   - 取消选中
   - 右侧恢复默认信息

## 11. 城市操作面板

### 11.1 资源信息区

显示该城市所属网络的四种资源：

1. 粮食
2. 木材
3. 矿石
4. 科技点

每项都必须是：

1. 对应颜色六边形图标
2. 图标周围无白边
3. 图标本体内部没有任何文字
4. 图标右侧有数值
5. 四项资源分两行排列并对齐

示意：

```text
+--------------------------------------+
|   __   12       __    8             |
|  /  \          /  \                 |
|  \__/          \__/                 |
|                                      |
|   __    5       __    9             |
|  /  \          /  \                 |
|  \__/          \__/                 |
+--------------------------------------+
```

正式版中六边形图标内部只有颜色，没有文字；它们本质上就是从地图里“摘出来”的同风格六边形。

### 11.2 科技区

城市操作面板还必须显示该网络的科技树：

1. `Agriculture`
2. `Logging`
3. `Mining`
4. `Education`

规则：

1. 已研究科技加粗显示。
2. 已研究科技点击无响应。
3. 未研究科技点击进入科研确认界面。

### 11.3 面板结构建议

```text
+--------------------------------------+
| [资源六边形区]                         |
|                                      |
| Tech:                                |
| * Agriculture                        |
|   Logging                            |
|   Mining                             |
| * Education                          |
|                                      |
```

## 12. 建筑建造流程界面

依据第 `19` 节，点击资源图标后不是直接建造，而是进入建筑确认界面。

### 12.1 进入方式

在城市操作面板中，点击某项资源图标。

资源图标和建筑对应关系来自 `5.4`：

1. 粮食 -> Farm
2. 木材 -> Lumber Mill
3. 矿石 -> Mine
4. 科技点 -> Library

### 12.2 建造确认界面

```text
+--------------------------------------+
|   __                                 |
|  /  \   count: 2                     |
|  \__/                                |
|                                      |
| Build                                |
| Cancel                               |
| -------------------------------      |
| 提示信息                              |
+--------------------------------------+
```

其中顶上一行至少显示：

1. 所选资源对应颜色图标
2. 所选城市里该建筑当前数量

### 12.3 成功与失败

1. 成功
   - 扣资源
   - 建造
   - 重算归属
   - 结算回合
   - 刷分
   - 进入下一回合或终局
2. 失败
   - 不结束回合
   - 停留在该确认界面
   - 显示失败原因

### 12.4 Cancel

1. 返回城市操作面板。
2. 保留当前选中的城市。

## 14. Skip 流程

依据第 `21` 节，`Skip` 永远在默认右侧面板中可见，点击后：

1. 不执行建造或研究。
2. 直接结算本回合。
3. 刷新分数。
4. 进入下一回合或终局。

## 15. Autoplay 模式的界面要求

依据第 `22` 节。

### 15.1 进入方式

1. 初始菜单点 `Autoplay`
2. 在地图选择界面配置：
   - `AI Type`
   - `Playback`
3. 点击 `Start`
4. 进入游戏界面

### 15.2 Normal

1. AI 逐回合执行。
2. 界面基本按回合同步刷新。
3. 用于观察过程。

### 15.3 Speed

1. 逻辑尽快连续跑。
2. 界面低频批量刷新。
3. 默认建议：
   - `100ms`
   - 或 `10` 个成功回合
   - 先到者触发刷新

### 15.4 Autoplay 下玩家仍可做的事

只允许：

1. `m`
   打开菜单
2. `q`
   退出

不允许：

1. 手动点击地图建造
2. 手动科研
3. 手动插入动作

因此 Autoplay 版游戏主界面中，右侧不应出现可误触的人类操作按钮。

```text
|  |                                           |   | STEP               |  |
|  |                                           |   | 01                 |  |
|  +-------------------------------------------+   | ------------------ |  |
|                                                  | mode: autoplay     |  |
|                                                  | ai: baseline       |  |
|                                                  | tip: <optional>    |  |
|                                                  +--------------------+  |
+--------------------------------------------------------------------------+
```

补充约束：

1. `SCORE` 与 `STEP` 的显示位置和风格保持与手动模式一致。
2. `mode` 与 `AI type` 放在右侧底部的小字信息区。
3. 若出现提示，则提示占用该区域的第二行或下一行小字位置。
4. Autoplay 游戏界面中不出现人工 `Build / Cancel / Research` 操作项。

## 19. 视觉规范

本节把第 `29` 节拆成更可执行的要求。

### 19.1 技术路线

正式方案固定为：

1. 主框架：`Textual`
2. 颜色与富文本：`Rich`
3. 高精度图像显示：优先采用 `textual-image` 一类图像后端
4. 图像生成：使用离屏光栅渲染生成 LOGO、数字、地图与资源图标

### 19.1.1 当前字符渲染路线的结论

在正式实现中，不应继续依赖“一个个字符 / Rich 段落 / 小按钮”去直接拼接：

1. 六边形地图
2. LOGO 六边形
3. 资源六边形图标
4. 小型数字
5. 大型数字

原因如下：

1. 终端字符网格不是像素网格，难以保证六边形真正严丝合缝。
2. 字符拼接方案很容易在不同尺寸下出现巨大空隙、比例漂移和断裂。
3. 数字阴影和弧度需要高精度控制，字符方案不稳定。
4. 地图、LOGO、资源图标若分散为大量字符控件，位置约束和对齐会持续失真。

因此当前路线应明确调整为：

1. 不再把“字符拼图”视为正式六边形或数字渲染方案。
2. `md` 中的字符图仅保留为结构示意，不再作为最终实现手段。

### 19.1.2 正式渲染策略

正式渲染应采用“离屏生成 + 图像显示”的混合方案：

1. 用统一的离屏渲染层生成高精度图像。
2. 用图像显示后端把结果显示到 Textual 界面中。
3. 不再让地图、LOGO、数字依赖单个字符单元的拼接精度。

建议的职责划分：

1. 离屏渲染层负责生成：
   - 主菜单 LOGO
   - 小型数字
   - 大型数字
   - 资源六边形图标
   - 整张地图
2. 图像显示层负责：
   - 在 Textual 中显示上述渲染结果
   - 处理尺寸、刷新和替换
3. 交互层负责：
   - 鼠标点击
   - 页面切换
   - 命中测试
   - 提示信息生命周期

### 19.1.3 `textual-image` 的位置

`textual-image` 或同类方案可以作为正式路线的一部分，但它的角色应明确：

1. 它适合作为“图像显示后端”。
2. 它不应承担“美术逻辑本身”。
3. 六边形、数字、LOGO 的外观仍应由项目自己的离屏渲染层决定。

换句话说：

1. 图像内容由项目自己生成。
2. `textual-image` 只负责把图像稳定显示到终端中。

这样做的原因是：

1. 我们需要完全掌握六边形拼接、数字弧度、阴影和选中边框。
2. 若只依赖终端字符 fallback，精度会重新下降。
3. 图像后端解决的是“怎么显示”，不是“显示成什么样”。

### 19.1.4 地图的正式策略

地图不应继续使用“每个格一个字符控件”的方案。

正式地图策略应为：

1. 整张地图作为一个整体渲染面输出。
2. 地图中的每个六边形在同一张画布上统一排布。
3. 城市、道路、普通地形只通过颜色差异表达。
4. 当前选中格的白边框也在同一张图上绘制。

这样才能保证：

1. 六边形之间真正紧密拼接。
2. 地图作为一个正方形主区域整体缩放。
3. 不会因为 widget 间距而把地图拆碎。

### 19.1.5 数字的正式策略

数字不应继续依赖“字符高度随布局自然伸缩”的方式。

正式数字策略应为：

1. 小型数字和大型数字分别拥有固定画布尺寸。
2. 每个数字采用固定模板。
3. 阴影方向固定为向右。
4. 数字显示时只在固定画布内切换内容，不改变整体外框。

这可以通过两种实现路径达成：

1. 模板光栅化
   - 推荐
   - 用代码定义数字模板，再渲染到图像上
2. 类似七段显示的受控区域切换
   - 可作为备选
   - 更适合非常规则、硬边的数字风格

考虑到当前目标数字带一定弧度，正式版更推荐“模板光栅化”，而不是纯七段矩形译码器风格。

### 19.1.6 文字的策略

普通说明文字仍可使用终端字体。

但以下内容若最终效果不稳定，也应允许切换到图像渲染：

1. `MicroCiv`
2. `SCORE`
3. `STEP`
4. 终局和地图选择界面的强化标题

原则是：

1. 普通文本追求可读性即可。
2. 强视觉标题追求统一风格与精确对齐。

### 19.2 点阵字的正式用途

必须优先美术化：

1. `MicroCiv`
2. `SCORE`
3. 分数字
4. `STEP`
5. 回合数字
6. 初始菜单、地图选择、终局界面的核心英文标题

普通说明文本仍可使用终端默认字体。

### 19.3 点阵字与阴影

应支持：

1. 主字形
2. 阴影层
3. 阴影偏移
4. 阴影色可配置

参考图 `number-like-this.png` 的核心特征是：

1. 大数字/大英文由规则点阵组成
2. 主体颜色明亮
3. 阴影方向固定为向右偏移
4. 阴影明显但不抢主体
5. 对于中空字形，阴影不仅出现在外轮廓右侧，也要同时出现在内轮廓对应位置

小型数字用于右侧信息区的普通数字显示，可先按下面的块面语言实现：

```text
 ███▒
█▒  █▒
█▒  █▒
 ███▒
```

大型数字用于更强调的分数或标题数字，可先按下面的比例实现：

```text
   █████▒▒
 ██▒▒   ██▒▒
██▒▒     ██▒▒
██▒▒     ██▒▒
██▒▒     ██▒▒
██▒▒     ██▒▒
 ██▒▒   ██▒▒
   █████▒▒
```

注意：

1. 阴影方向固定，不向下偏移，只沿轮廓右侧延伸。
2. 若字形内部有空腔，例如 `0 / O / D / P / R`，内轮廓也要遵守同一“右侧阴影”规则。
3. 阴影必须贴着数字或字母本身的轮廓走，不能脱离轮廓单独多出一整列阴影块。
4. 其他数字依此类推，先做出统一语言，后续再微调弧度、宽度和拐角。
5. 这一要求适用于：
   - 分数字
   - 回合数字
   - 大标题字

### 19.4 LOGO 规范

依据 `logo.jpg`：

1. LOGO 由 7 个六边形图标组成。
2. 是“七种地形对应的六边形组合”。
3. 正式实现不应保留图片里的黑色手绘边框。
4. 重点是颜色块和六边形拼接关系。
5. 每个六边形内部只用颜色块，不放文字。
6. LOGO 六边形本体也不带默认边框。

其组合关系可简记为：

```text
          [ A ]

     [ B ] [ C ] [ D ]

         [ E ] [ F ]

           [ G ]
```

也可近似成：

```text
                   __
                __/  \__
               /  \__/  \
               \__/  \__/
               /  \__/  \
               \__/  \__/
                  \__/
```

### 19.5 六边形图标复用要求

同一套六边形风格不仅用于 LOGO，还应复用于：

1. 地图格子外形
2. 城市操作面板中的资源图标

也就是说：

1. LOGO 不是独立美术孤岛
2. 地图六边形
3. 资源六边形

三者应使用同一视觉语言

并进一步约束：

1. 地图格内部只用颜色表达信息。
2. 资源六边形图标内部也只用颜色表达信息。
3. LOGO、地图格、资源图标默认都没有边框。
4. 不在六边形内部塞任何字母、数字或覆盖物标记。
5. 唯一例外是地图当前选中格，它需要额外出现白色边框。

### 19.6 推荐模块划分

TUI 层应至少拆成：

1. 主应用模块
2. 页面状态机模块
3. 地图渲染模块
4. 右侧面板模块
5. LOGO 渲染模块
6. 点阵字模块
7. Records 视图模块
8. Autoplay 刷新控制模块

## 20. 当前实现需要被纠正的偏差

结合本规范，当前 TUI 后续重做时必须优先修正这些方向性问题：

1. 初始菜单不应出现 `Baseline / Normal / Speed` 的概念。
2. 地图选择界面才是地图参数和 Autoplay 专属配置的承载页。
3. 游戏主界面默认右侧不应常驻大型状态面板。
4. 人类操作项不应一次性全部展开成可滚动长列表。
5. Autoplay 下不应继续显示人工操作按钮。
6. 地图区域必须成为最大视觉主体。
7. 地图区应为几乎占满屏幕高度的正方形区域，左侧底部不留提示区。
8. 地图格内部必须只有颜色，不得放字母、数字或覆盖物图标。
9. 资源信息区必须使用纯颜色六边形图标，不是普通文字表。
10. 游戏内正常操作不应依赖滚轮，滚轮只用于 Records。
11. Autoplay 下右侧底部小字区应显示模式与 AI 类型，提示出现时占下一行。
12. Records 无记录时不显示 Export。
13. Records 列表必须回到左右两列且铺满宽度。
14. 终局页应以“左侧大正方形地图、右侧分数与三个选项”为主，而不是普通摘要卡片。
15. 全屏终端下界面必须扩张利用空间，而不是只占中心一小块。

## 22. 结论

阶段一 TUI 的正式目标不是“有按钮能点就算完成”，而是：

1. 页面流程严格符合 `microciv.md`
2. 地图、右侧面板和 Records 布局严格符合 `microciv.md`
3. LOGO、六边形图标、分数与回合数字符合第 `29` 节的统一视觉路线
4. 全屏终端下合理利用空间
5. Play 与 Autoplay 在同一逻辑体系下表现不同界面权限，而不是拼接出两个互相打架的 UI

后续实际重构 TUI 时，应以本文档和 `docs/microciv.md` 为准，而不是以当前已有 TUI 实现为准。
