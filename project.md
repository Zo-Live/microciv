# MicroCiv 阶段一执行冻结规范

## 1. 文档定位

本文件是阶段一的实现冻结文档，用于补齐 [docs/microciv.md](docs/microciv.md) 中未量化、未定序或未明确数据口径的部分。

执行原则如下：

1. `docs/microciv.md` 仍是需求背景文档
2. 本文件负责把阶段一实现所需的留白补成可直接编码的标准
3. 若两者在阶段一实现细节上冲突，以本文件为准
4. 本文件不修改阶段一主规则框架，只补齐默认值、阈值、顺序和接口边界

## 2. 阶段一交付边界

阶段一必须实现以下内容：

1. 完整的规则逻辑层
2. 可复现的地图生成
3. 可运行的 Textual TUI 主流程
4. 手动 Play 模式
5. Autoplay 的 Baseline AI
6. 本地持久化的 Records
7. CSV 导出

阶段一不实现以下内容：

1. Expert AI 的可运行逻辑
2. Custom AI 的可运行逻辑
3. LLM 接口接入
4. 阶段二搜索器

阶段一 UI 约束如下：

1. `Autoplay` 入口只允许启动 `Baseline`
2. 地图选择界面不展示可启动的 `Expert` 与 `Custom`
3. 阶段二相关枚举、接口和记录字段保留，但不提供可运行按钮

## 3. 默认配置与全局常量

### 3.1 默认配置

1. 默认地图大小 `map_size = 6`
2. 默认总回合数 `turn_limit = 80`
3. 默认地图难度 `map_difficulty = normal`
4. 默认游戏模式 `mode = play`
5. `Play` 模式默认 `policy_type = none`
6. `Play` 模式默认 `playback_mode = none`
7. 进入 `Autoplay` 时默认 `policy_type = baseline`
8. 进入 `Autoplay` 时默认 `playback_mode = normal`
9. 默认游戏版本读取 `pyproject.toml` 中的 `project.version`，阶段一初始值为 `0.1.0`

### 3.2 播放与刷新常量

1. Autoplay `Normal` 模式的目标展示间隔为 `300ms`
2. Autoplay `Speed` 模式的刷新阈值为“距离上次刷新 `100ms` 或累计 `10` 个成功回合，先到者触发”
3. 进入终局、按下 `m`、按下 `q` 时必须立即强制刷新

### 3.3 数值常量

1. 覆盖收益固定为 `8`
2. 河流修路额外代价固定为：优先 `3 木材`，否则 `2 矿石`，否则失败
3. 每座城市每回合粮食消耗固定为 `4`
4. 每座城市建筑上限固定为 `6`

## 4. 坐标、排序与稳定顺序

### 4.1 地图坐标

1. 地图使用轴向坐标 `(q, r)`
2. 第三个隐含坐标定义为 `s = -q-r`
3. 合法格满足 `max(abs(q), abs(r), abs(s)) <= map_size - 1`
4. 地图总格数公式为 `3a(a-1)+1`，其中 `a = map_size`

### 4.2 六方向邻接顺序

全局固定六个邻接方向，顺序如下：

1. `(1, 0)`
2. `(1, -1)`
3. `(0, -1)`
4. `(-1, 0)`
5. `(-1, 1)`
6. `(0, 1)`

所有邻接遍历、BFS、路径搜索、坐标 tie-break 都必须复用该顺序。

### 4.3 坐标字典序

文档中所有“坐标字典序更小”统一定义为：

1. 先比较 `q`
2. 若 `q` 相同，再比较 `r`
3. 更小者优先

## 5. 逻辑层数据建模标准

### 5.1 地图格建模

每个地图格必须保留两层信息：

1. `base_terrain`
   可取 `plain / forest / mountain / river / wasteland`
2. `occupant`
   可取 `none / city / road`

阶段一禁止把建造后的城市或道路直接覆写掉原始地形信息。

1. 城市 `city_id`
2. 道路 `road_id`
3. 城市网络 `network_id`
4. 对局记录 `record_id`

ID 规则统一为：

1. 每局游戏内从 `1` 开始递增
2. 同类对象 ID 不复用
3. Records 的 `record_id` 在持久化文件内全局递增

### 5.3 城市数据

每座城市必须至少保存：

1. `city_id`
2. `coord`
3. `founded_turn`
4. `network_id`
5. 四类建筑数量
6. 建筑总数

### 5.4 道路数据

每条道路必须至少保存：

1. `road_id`
2. `coord`
3. `built_turn`

### 5.5 城市网络数据

每个网络必须至少保存：

1. `network_id`
2. 城市集合
3. 共享资源池
4. 已解锁科技集合

网络合并规则补充如下：

1. 合并后保留 `network_id` 最小的网络作为存续网络
2. 其他网络并入后删除
3. 合并后的资源池为各网络资源逐项求和
4. 合并后的科技集为并集

## 7. 地图生成冻结方案

### 7.1 总体原则

地图生成必须满足：

1. 同一组 `GameConfig` 中的地图参数与 `seed` 生成完全一致的初始地图
2. 不允许使用不带种子的系统随机源
3. 地图生成中所有 tie-break 都必须由固定顺序或同一 RNG 决定

### 7.2 区域数量

设总格数为 `cell_count`，区域数量 `region_count` 计算为：

`region_count = clamp(round(cell_count / 18), 3, 8)`

### 7.3 区域种子选择

区域划分使用“贪心分散种子 + 多源 BFS”：

1. 第一个种子从全部合法格中按 seeded RNG 选取
2. 后续种子每次选择“到已有种子最短 hex 距离最大”的候选格
3. 若最远候选有多个，按 seeded RNG 从这些最远候选中选一个
4. 选完全部种子后，执行多源 BFS 进行区域归属

### 7.4 区域归属

多源 BFS 的具体规则如下：

1. 初始队列按种子加入顺序入队
2. 按固定邻接顺序扩张
3. 每个格只归属于第一个到达它的区域
4. 区域必须保持连通

### 7.5 普通地形目标比例

以下比例指“河流叠加前”的普通地形目标占比。

`Normal`：

1. 平原 `30%`
2. 森林 `25%`
3. 山地 `25%`
4. 荒地 `20%`

`Hard`：

1. 平原 `20%`
2. 森林 `25%`
3. 山地 `30%`
4. 荒地 `25%`

### 7.6 主地形分配

每个区域主地形从 `plain / forest / mountain / wasteland` 中选择，规则如下：

1. 优先满足全图目标比例
2. 若存在满足目标比例余额的候选主地形，则优先选择“不与任何相邻区域主地形重复”的候选
3. 只有在所有满足比例余额的候选都与相邻区域重复时，才允许相邻区域主地形重复
4. 全图必须至少存在一个 `plain` 主导区域
5. 每个区域主地形占该区域最终普通地形格数的 `70% ~ 85%`

### 7.7 次级地形斑块

次级地形生成冻结为以下规则：

1. 每区域注入 `1 ~ 3` 个次级斑块
2. 每个斑块大小 `2 ~ 5` 格
3. 次级地形只能从其他普通地形中选取，不含河流
4. 斑块使用局部 BFS 生长，不允许逐格独立随机散点
5. 一个区域最多混入 `2` 种次级地形

### 7.8 邻域平滑

邻域平滑最多执行 `2` 轮，每轮规则如下：

1. 仅处理普通地形，不处理河流
2. 若某格在 6 邻域中与自身同类邻居数 `<= 1`，则改为其邻域多数地形
3. 若出现平票，则保持原地形
4. 平滑不得把区域主地形占比压到 `70%` 以下

### 7.9 河流条数与长度

河流生成冻结为以下规则：

1. `Normal` 固定生成 `1` 条河流
2. `Hard` 在 `map_size <= 6` 时生成 `1` 条河流，在 `map_size >= 7` 时生成 `2` 条河流
3. 单条河流必须是连续、自避免路径
4. 默认不生成分叉

单条河流最小长度要求如下，单位为“河流格数”：

1. `Normal` 不得小于 `ceil(2 * (map_size - 1) * 0.6)`
2. `Hard` 不得小于 `ceil(2 * (map_size - 1) * 0.75)`

### 7.10 河流路径生成

河流采用“边界到边界的自避免路径”：

1. 起点必须位于边界格
2. 终点必须位于另一侧边界格
3. 终点候选优先选择与起点 hex 距离更远的边界格
4. 每步从未访问邻格中选择
5. 下一步候选按 `distance_to_end + 0.25 * turn_penalty` 排序，其中直行 `turn_penalty = 0`，转向 `turn_penalty = 1`
6. 若排序并列，则按 seeded RNG 打破并列
7. 若路径死路或长度不达标，则重试本条河流
8. 单张地图河流生成最多重试 `20` 次，仍失败则整图重生成

### 7.11 河流与荒地修补

河流叠加后立即执行：

1. 所有与河流相邻的荒地必须改成非荒地
2. 若该格所在区域主地形不是荒地，则改回区域主地形
3. 否则改为平原
4. 若该格改为平原会导致平原占比超过当前难度目标平原占比 `+5` 个百分点，则改为森林

### 7.12 平原保底

若整图无平原，则：

1. 从非河流格中选取一个坐标字典序最小的合法格
2. 将其改为平原

### 7.13 质量检查阈值

地图质量检查冻结如下。

`Normal`：

1. 可建城地形占比不得低于 `55%`
2. 平原占比不得低于 `15%`
3. 荒地占比不得高于 `25%`
4. 最大荒地连通块不得超过总格数的 `18%`

`Hard`：

1. 可建城地形占比不得低于 `45%`
2. 平原占比不得低于 `10%`
3. 荒地占比不得高于 `33%`
4. 最大荒地连通块不得超过总格数的 `25%`

通用约束：

1. 任意单条河流长度不得超过总格数的 `40%`
2. 普通地形中任一种占比不得超过 `45%`

### 7.14 质量修补与重生成

质量检查未通过时：

1. 先执行 `1` 轮局部修补
2. 局部修补只允许把“过量地形”改写为“短缺地形”
3. 局部修补不得创建与河流相邻的荒地
4. 局部修补后重新检查
5. 若仍不通过，则整图重生成

## 8. 建城、修路、覆盖与网络规则冻结

### 8.1 建城与修路

建城、修路的合法性仍按 `docs/microciv.md`，补充以下口径：

1. “相邻”统一指六边形 1-ring 邻接
2. 新城建成后若与一个或多个网络通过城市、道路、河流连通，则当场并网
3. 新路建成后若连接多个网络，则当场并网

### 8.2 覆盖收益

覆盖收益口径固定为：

1. 覆盖平原：立刻获得 `+8 粮食`
2. 覆盖森林：立刻获得 `+8 木材`
3. 覆盖山地：立刻获得 `+8 矿石`
4. 覆盖河流：立刻获得 `+8 粮食` 和 `+8 科技点`
5. 覆盖荒地：无收益

### 8.3 河流修路扣费

河流上修路的额外扣费顺序固定为：

1. 若所选支付网络木材 `>= 3`，扣 `3` 木材
2. 否则若矿石 `>= 2`，扣 `2` 矿石
3. 否则失败

### 8.4 多网络相邻时的扣费网络选择

若待修道路格与多个独立网络相邻，则固定按以下顺序选择支付网络：

1. 先取能支付额外过河成本的网络
2. 在这些网络中，优先选择“最早建城时间最早”的网络
3. 若并列，优先选择“该网络中坐标字典序最小城市”更小的网络
4. 若仍并列，优先选择 `network_id` 更小的网络

## 9. 资源归属、停工与结算规则冻结

### 9.1 资源归属

资源格归属规则补充如下：

1. 河流的粮食归属和科技点归属分别独立判定
2. 同一个河流格的粮食与科技点允许归属于不同城市
3. 每次成功动作后，在本回合资源结算前重算全部资源格归属

### 9.2 归属 tie-break

资源格归属的比较顺序固定为：

1. 先比较该资源对应建筑数量
2. 若并列，比较建城时间，建城更早者优先
3. 若并列，比较城市坐标字典序，更小者优先
4. 若仍并列，比较 `city_id`，更小者优先

### 9.3 结算顺序

每个成功回合的结算顺序严格固定为：

1. 执行动作
2. 必要时合并网络
3. 重算全部资源格归属
4. 以“回合开始结算快照”判定各网络是否缺粮停工
5. 结算全部城市地形资源产出
6. 结算全部城市建筑产出
7. 按城市数量结算粮食消耗
8. 刷新当前分数
9. 推进回合或进入终局

### 9.4 缺粮停工

缺粮停工口径固定为“网络级快照”：

1. 在步骤 4 取每个网络的粮食快照
2. 若某网络快照粮食 `<= 0`，则该网络本回合处于停工状态
3. 停工网络中的城市本回合仅保留粮食产出
4. 停工网络中的木材、矿石、科技点地形产出全部为 `0`
5. 停工网络中的伐木场、矿场、图书馆产出全部为 `0`
6. 停工网络中的农场产出仍然生效
7. 停工判定不受城市遍历顺序影响

### 9.5 当回合即时生效规则

以下效果在动作成功的当回合立即参与结算：

1. 新建城市可在当回合参与资源归属与产出
2. 新建道路可在当回合提供网络连通
3. 新建建筑可在当回合提供其建筑产出
4. 新研究科技只解锁后续建造权限，本回合不直接产生资源

## 11. Records 与导出标准

### 11.1 保存时机

阶段一只保存“自然完成到终局”的对局记录。

以下情况不写入 Records：

1. 中途返回菜单
2. 中途退出程序
3. 程序异常中止

### 11.2 存储位置

阶段一采用仓库根目录相对路径：

1. 本地持久化文件：`data/records.json`
2. 导出目录：`exports/`

若目录不存在，程序启动时自动创建。

### 11.3 内部持久化格式

内部主存储使用 JSON，顶层结构固定为：

```json
{
  "schema_version": 1,
  "next_record_id": 1,
  "records": []
}
```

## 13. Baseline AI 冻结策略

### 13.1 总要求

Baseline 必须满足：

1. 任意合法状态都返回合法动作
2. 不允许纯随机落子
3. 所有 tie-break 必须使用固定顺序，不允许运行期随机
4. 当不存在其他可执行优先动作时，回退 `skip`

### 13.2 动作优先级

Baseline 的全局动作优先级固定为：

1. 粮食补救
2. 连网修路
3. 高收益建城
4. 关键科技
5. 高缺口建筑
6. 其他合法建筑
7. `skip`

### 13.3 粮食危险判定

若任一网络满足以下任一条件，则视为粮食危险网络：

1. 当前粮食 `< 8`
2. 按当前状态执行一次 `skip` 结算的预测结果中，回合结束粮食 `< 4`

出现粮食危险网络时，Baseline 必须优先考虑能提升该网络粮食安全的动作。

### 13.4 建城评分

Baseline 建城评分固定按候选格 1-ring 资源潜力计算：

`score = 3 * food + 2 * wood + 2 * ore + 1 * science`

其中：

1. 平原记 `food += 3`
2. 森林记 `wood += 2`
3. 山地记 `ore += 2`
4. 河流记 `food += 2` 且 `science += 1`
5. 荒地不计分

若多个候选得分相同，则依次按以下顺序选择：

1. 更高的 `food`
2. 更高的 `science`
3. 更小的坐标字典序

### 13.5 修路策略

Baseline 修路规则固定为：

1. 对每个合法修路动作计算 `road_score`
2. `road_score = danger_bonus + merge_bonus + terrain_bonus - cost_penalty`
3. `danger_bonus = 120`，当该道路能让粮食危险网络与当前粮食 `>= 8` 的其他网络连通时取该值，否则为 `0`
4. `merge_bonus = 60`，当该道路能合并任意两个现有网络时取该值，否则为 `0`
5. `terrain_bonus = 3 * adjacent_plain_count + 2 * adjacent_river_count`
6. `cost_penalty = 4`，当该道路建在河流上时取该值，否则为 `0`
7. 选择 `road_score` 最高的候选
8. 若并列，优先选择坐标字典序更小者

### 13.6 科技策略

科技优先级固定为：

1. 若缺乏粮食增产且能研究 `Agriculture`，优先研究 `Agriculture`
2. 若已有稳定粮食且科技点偏低，优先研究 `Education`
3. 若木材长期不足，优先研究 `Logging`
4. 若矿石长期不足，优先研究 `Mining`

其中：

1. “已有稳定粮食”定义为当前粮食 `>= 12`，且按当前状态执行一次 `skip` 结算后粮食仍 `>= 8`
2. “科技点偏低”定义为当前科技点 `< 10`，且按当前状态执行一次 `skip` 结算时科技点净增长 `<= 1`

“长期不足”统一定义为：

1. 当前资源 `< 该资源对应下一座建筑成本`
2. 且最近 `3` 回合预计平均净增量 `<= 0`

### 13.7 建筑策略

建筑优先级按资源缺口从高到低处理：

1. 粮食缺口最大时优先农场
2. 木材缺口最大时优先伐木场
3. 科技点缺口最大时优先图书馆
4. 矿石缺口最大时优先矿场

缺口评分定义为：

`gap_score = target_buffer - current_resource`

目标缓冲固定为：

1. 粮食 `12`
2. 木材 `10`
3. 矿石 `8`
4. 科技点 `10`

当确定要造某类建筑时，城市选择顺序固定为：

1. 优先选择与该建筑对应资源格相邻数量更多的城市
2. 若并列，优先选择该类建筑现有数量更少的城市
3. 若并列，优先选择建城时间更早的城市
4. 若并列，优先选择坐标字典序更小的城市

### 13.8 Baseline 可用性验收

阶段一 AI 验收目标冻结为：

1. 在默认配置下，对固定 `30` 个测试 seed 运行完整对局
2. Baseline 平均分必须高于随机策略平均分
3. Baseline 在至少 `20 / 30` 个 seed 上得分不低于随机策略

7. 新城、新路、新建筑当回合立即生效

至少覆盖以下断言：

1. 失败动作不推进回合

3. 第 `T` 回合成功结算后直接进入终局
4. 不显示第 `T + 1` 回合

1. 对局完成后写入 `data/records.json`
